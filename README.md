# Smart Campus API

## Project Overview

This project implements a RESTful API for a Smart Campus system that manages rooms, sensors, and sensor readings. The system simulates a real-world campus environment where physical spaces are equipped with sensors that continuously generate data.

The API is designed using JAX-RS and follows REST architectural principles, including resource-based design, stateless communication, and the use of standard HTTP methods and status codes. The implementation also incorporates validation, exception handling, and logging to ensure robustness and maintainability.

## Objectives

The main objectives of this project are:

* To design a RESTful API using JAX-RS
* To model real-world entities such as rooms and sensors
* To implement relationships between resources
* To enforce business rules through validation
* To handle errors using structured exception mapping
* To demonstrate best practices in API development

## System Architecture

The system is structured around three core resources:

### Room

Represents a physical location within the campus.

* Each room has a unique ID, name, and capacity
* A room can contain multiple sensors

### Sensor

Represents a monitoring device installed in a room.

* Each sensor has a type (e.g., temperature)
* Sensors are linked to a specific room
* Each sensor maintains its current value

### Sensor Reading

Represents a data record generated by a sensor.

* Contains timestamp and measured value
* Each sensor can have multiple readings

## Resource Relationships

The system maintains relationships between resources:

* A **Room contains multiple Sensors**
* A **Sensor belongs to one Room**
* A **Sensor generates multiple Readings**

This relationship is implemented using identifiers, ensuring a clear and scalable design.

## API Endpoints

### Room Endpoints

| Method | Endpoint    | Description                              |
| ------ | ----------- | ---------------------------------------- |
| GET    | /rooms      | Retrieve all rooms                       |
| POST   | /rooms      | Create a new room                        |
| GET    | /rooms/{id} | Retrieve a specific room                 |
| DELETE | /rooms/{id} | Delete a room (only if no sensors exist) |

### Sensor Endpoints

| Method | Endpoint                  | Description                      |
| ------ | ------------------------- | -------------------------------- |
| GET    | /sensors                  | Retrieve all sensors             |
| GET    | /sensors?type=Temperature | Filter sensors by type           |
| POST   | /sensors                  | Create a sensor linked to a room |

### Sensor Reading Endpoints

| Method | Endpoint               | Description           |
| ------ | ---------------------- | --------------------- |
| GET    | /sensors/{id}/readings | Retrieve all readings |
| POST   | /sensors/{id}/readings | Add a new reading     |

## Validation and Business Rules

The system enforces several important rules:

* A sensor cannot be created unless the referenced room exists
* A room cannot be deleted if sensors are assigned to it
* A sensor cannot accept readings when in "MAINTENANCE" state

These rules ensure data integrity and reflect real-world constraints.

## Error Handling

The API implements structured exception handling using custom exceptions and exception mappers.

| Scenario                 | Status Code               | Description           |
| ------------------------ | ------------------------- | --------------------- |
| Room has sensors         | 409 Conflict              | Prevents deletion     |
| Invalid room reference   | 422 Unprocessable Entity  | Invalid input data    |
| Sensor under maintenance | 403 Forbidden             | Operation not allowed |
| Unexpected error         | 500 Internal Server Error | Generic fallback      |

All error responses are returned in JSON format for consistency.

## Logging

A logging filter is implemented to track:

* Incoming requests (method and URI)
* Outgoing responses (status codes)

This helps in debugging and monitoring API activity.

## Sample curl Commands

### Create Room

```bash
curl -X POST http://localhost:8080/api/v1/rooms \
-H "Content-Type: application/json" \
-d '{"id":"ROOM-1","name":"Lecture Hall","capacity":100}'
```

### Create Sensor

```bash
curl -X POST http://localhost:8080/api/v1/sensors \
-H "Content-Type: application/json" \
-d '{"id":"TEMP-001","type":"Temperature","status":"ACTIVE","currentValue":25,"roomId":"ROOM-1"}'
```

### Add Sensor Reading

```bash
curl -X POST http://localhost:8080/api/v1/sensors/TEMP-001/readings \
-H "Content-Type: application/json" \
-d '{"id":"R1","timestamp":1710000000,"value":27.5}'
```

### Get All Rooms

```bash
curl http://localhost:8080/api/v1/rooms
```

### Delete Room

```bash
curl -X DELETE http://localhost:8080/api/v1/rooms/ROOM-1
```

## Limitations

* The system uses in-memory storage, so data is not persistent
* No authentication or authorization is implemented
* Designed for demonstration purposes only
